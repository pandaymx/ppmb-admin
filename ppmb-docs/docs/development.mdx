# 开发规约

为了确保 PPMB 系统的高质量演进，我们在日常开发中遵循着严格的代码规范与测试约束。

## 代码质量红线 (SonarQube)

项目通过 SonarQube / SonarCloud 进行自动化持续集成代码检查。开发者提交的代码必须符合如下标准（部分核心规则）：

*   **异常声明规约**:
    *   禁止在方法的 `throws` 子句中声明不必要的异常 (java:S1130)。如果方法体内没有任何代码会抛出声明中的受检异常，或者异常类是多余的（例如已声明父类异常），则必须将其移除。
    *   禁止在单元测试方法上声明不需要的 `throws Exception`，以保证测试代码的紧凑、整洁。
*   **消除技术债**:
    *   必须定期扫描并修复报告中的 Code Smell。
    *   优先级聚焦：阻碍（Blocker）和严重（Critical）级别的问题被视为上线红线，必须优先处理和清零。

## 测试先行 (Test-Driven Development)

我们推崇 TDD（测试驱动开发）的工程实践，通过前置测试来提升代码鲁棒性和重构安全性：

*   **硬性约束**: 新增功能或修复 Bug 必须同步提供关联的单元测试。
*   **代码覆盖率**: 提交的代码需通过 CI Pipeline 的覆盖率要求检查；未达标的代码将被阻塞合并（Block Merge）。
*   **中间件测试要求**: 禁止对 Redis 交互操作（如限流、锁）使用 Mock 对象进行测试；须采用 `Testcontainers` (e.g., redis:7-alpine) 结合集成测试以真正验证并发表现及 Lua 脚本正确性。

## 通用编码准则

*   **数据载体**: 对于 DTO 或 Value Object 优先使用 Java 21+ 引入的 `record` 类型，保持属性不可变。
*   **异常响应**: 强制使用自定义异常，抛出错误时遵循 Spring Boot 4.0 的 `ProblemDetails` 标准。
*   **Git 提交**: 遵守 Conventional Commits 规范，统一中文提交 ` <type>(<scope>): <subject> `。
